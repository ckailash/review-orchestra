How it works

The CLI returns a ReviewResult JSON on stdout. Here is a simplified example:

// ReviewResult (simplified) { "sessionId": "20260315-143022", "round": 1, "findings": [ { "id": "f-001", "title": "SQL injection in user lookup", "file": "src/auth/users.ts", "line": 42, "confidence": "verified", "impact": "critical", "severity": "p0", "source": ["claude", "codex"], "preExisting": false }, { "id": "f-002", "title": "Missing null check on response.data", "file": "src/api/client.ts", "line": 87, "confidence": "likely", "impact": "functional", "severity": "p1", "source": ["claude"], "preExisting": false }, { "id": "f-003", "title": "Variable name 'x' is not descriptive", "file": "src/utils/parse.ts", "line": 15, "confidence": "possible", "impact": "nitpick", "severity": "p3", "source": ["codex"], "preExisting": false } ], "comparison": { "new": 3, "persisting": 0, "resolved": 0 } }

Features

Supervised Flow

You control every step. See findings, pick what to fix, confirm actions, re-review when ready. No autonomous changes.

1 Review → 2 Present → 3 Decide → 4 Fix → 5 Re-review

Severity Classification

Two-axis scoring: confidence (Verified / Likely / Possible / Speculative) and impact (Critical / Functional / Quality / Nitpick) produce P0-P3 levels.

CriticalFunctionalQualityNitpick
VerifiedP0P1P2P3
LikelyP0P1P2P3
PossibleP1P2P3P3
SpeculativeP2P3P3P3

Fuzzy Dedup

Same bug, different words. Jaccard similarity on tokenised titles catches cross-reviewer duplicates and merges them into a single finding.

// Claude says: "SQL injection in auth.ts:42" // Codex says: "Unsanitised input in auth.ts line 42" // Consolidator output: merged → source: ["claude", "codex"]

Cross-Round Comparison

Tracks findings across review rounds, even through renames, shifted line numbers, and refactors. Resolved findings are detected automatically.

// Round 2 comparison output: { "new": 1, "resolved": 2, "persisting": 3 }

Session Persistence

Reviews survive across CLI invocations. Worktree hashes detect when files change between rounds. Sessions auto-expire when the base branch moves.

// .review-orchestra/session.json { "sessionId": "20260315-143022", "status": "active", "currentRound": 2, "worktreeHash": "abc123..." }

Natural Language Args

No flags to memorise. Say what you want in plain English and the CLI parses intent from natural language arguments.

/review-orchestra only use claude /review-orchestra fix quality issues too /review-orchestra skip codex /review-orchestra src/auth/ src/api/

Usage

Skill invocation (Claude Code)

claude code skill
# auto-detect scope, review everything
/review-orchestra
# review specific paths only
/review-orchestra src/auth/ src/api/
# extend threshold to P2 (include quality issues)
/review-orchestra fix quality issues too
# single reviewer mode
/review-orchestra only use claude
# disable a specific reviewer
/review-orchestra skip codex

CLI subcommands

CommandDescription
review-orchestra reviewRun reviewers + consolidate, return findings JSON. Default when no subcommand given.
review-orchestra review src/services/Review with scope args (paths, natural language filters).

Supervised workflow

The CLI runs review + consolidation and returns ReviewResult JSON. The skill then enters the supervised loop:

supervised workflow
# Step 1: Review (spawns Claude + Codex, consolidates)
/review-orchestra
# Step 2: Skill presents findings grouped by severity
P0 SQL injection in auth/users.ts:42 [claude, codex]
P1 Missing null check in api/client.ts:87 [claude]
P2 Unused import in utils/helpers.ts:3 [codex]
# Step 3: You decide what to fix
"fix the P0 and P1 findings"
# Step 4: Skill confirms planned actions, you approve
# Step 5: Orchestrator Claude fixes code via Edit/Write
# Step 6: Re-review to verify fixes
/review-orchestra
0 new | 0 persisting | 2 resolved
# Step 7: Done - commit, push, PR

Configuration

Create .review-orchestra/config.json in your project root for per-project config, or ~/.review-orchestra/config.json for global user defaults. Project overrides global. All settings below are optional — defaults are sensible.

Default config

{ "reviewers": { "claude": { "enabled": true, "command": "claude -p - --allowedTools \"Read,Grep,Glob,Bash\" --output-format json", "outputFormat": "json" }, "codex": { "enabled": true, "command": "codex exec - --output-last-message {outputFile} --json", "outputFormat": "json" } }, "thresholds": { "stopAt": "p1" } }

p0Critical onlyVerified/Likely + Critical impact. Security holes, data loss.
p1Critical + FunctionalDefault. Bugs that will break behaviour in production.

Finding comparison

Cross-round finding comparison uses LLM semantic matching by default (via Claude Haiku). Handles renamed files, shifted line numbers, and reworded descriptions. Falls back to heuristic matching on failure.

SettingDefaultDescription
method"llm"Comparison method: "llm" for semantic matching, "heuristic" for file+title matching
model"claude-haiku-4-5"Model to use for LLM comparison
timeoutMs60000Timeout for LLM comparison call (ms)
fallback"heuristic"Fallback method when LLM fails or times out

See Custom reviewers below for adding your own reviewer via config.

Session artifacts

All round data is stored in .review-orchestra/ in the project root. Everything is kept. Never auto-deleted.

Directory structure
.review-orchestra/ ├── session.json # Session state (rounds, reviews, reviewerErrors) ├── state.lock # PID file — concurrent-run prevention ├── progress.json # Live reviewer status (deleted on round complete) ├── round-1-claude-raw.txt # Raw stdout, persisted before parse ├── round-1-codex-raw.txt # Renamed to *.failed on codex subprocess failure └── round-2-claude-raw.txt
session.json
{ "sessionId": "20260315-143022", "status": "active", "scope": { "type": "branch", "baseBranch": "main", "description": "branch feat/auth vs main" }, "currentRound": 2, "worktreeHash": "abc123", "rounds": [ { "number": 1, "phase": "complete", "reviews": { "claude": { "..." } }, "consolidated": ["..."], "reviewerErrors": [], "worktreeHash": "def456" } ] }

Key fields

FieldDescription
sessionIdTimestamp-based unique ID (e.g. 20260315-143022)
statusactive (accepting rounds), expired (scope base changed, requires reset), completed (user ended session)
scopeDiff scope at session creation (uncommitted, branch, commit). Detached HEAD produces baseBranch: "detached@<sha7>" rather than the literal "HEAD" so cross-round comparison stays stable.
worktreeHashSHA-256 over HEAD + staged + unstaged + untracked files. Per-round snapshot for stale detection.
rounds[]Per-round: phase, reviews (per-reviewer findings + metadata), consolidated, reviewerErrors (preserved across crash recovery), findingsPersisted, timestamps.

Stale detection

Run review-orchestra stale to check if the worktree has changed since the last review. Exit codes:

Exit codeMeaning
0Fresh. Worktree matches last review round.
1Stale. Files changed since last review.
2No session. No previous review to compare against.

Add .review-orchestra/ to .gitignore. The setup command does this automatically.

Concurrent runs

Two terminals running review-orchestra review in the same project would trample each other's session. state.lock prevents this — it stores the holder's PID and uses an atomic-rename release protocol with a post-rename PID re-check, so a recovering process can't accidentally delete another instance's lock. Stale locks (PID gone) are detected via process.kill(pid, 0) and overwritten safely.

Crash recovery

If a review invocation crashes mid-round, the next invocation reads session.json, finds the round in phase: "reviewing" or phase: "consolidating", and resumes:

  • reviewing — re-runs only the reviewers not present in round.reviews, then merges with saved findings. A flake on the remaining reviewer no longer discards the saved work.
  • consolidating — skips reviewers entirely, restores reviewerErrors from the round, re-runs consolidation. The findingsPersisted flag prevents double-writes to the cross-session JSONL store.

Custom reviewers

Any CLI tool that accepts a prompt and produces output can be a reviewer. Two approaches:

TypeScript interface (src/reviewers/types.ts)
interface Reviewer { name: string; review( prompt: string, scope: DiffScope, context: ReviewerCallContext ): Promise<{ findings: Finding[]; rawOutput: string; elapsedMs?: number; }>; }
Config-based (config/default.json)
{ "reviewers": { "my-linter": { "enabled": true, "command": "my-tool review \"{prompt}\"", "outputFormat": "json" } } }

Config-based reviewers use the GenericReviewer class, which executes the command template (replacing {prompt}) and parses the output. The consolidator normalises all reviewer output into the standard findings schema regardless of source format. Reviewers persist their raw stdout to round-N-<name>-raw.txt via persistRawOutput() before parsing — debug evidence survives parse failures and reviewer crashes.
